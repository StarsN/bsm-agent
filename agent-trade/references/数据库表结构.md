# 数据库表结构

## 核心表

| 表 | 用途 | 谁写 |
|----|------|------|
| agent_candidates | 每轮"合约扫描与操作建议"快照（面板同源） | worker |
| token_heat_history | 热度历史（compute_composite_scores 用） | worker |
| market_snapshots | 合约快照（snapshot + analysis JSON） | worker |
| trade_positions | 当前持仓和已平仓历史 | auto_trader / trade_logic |
| trade_loss_archive | 机器止损归档 | trade_logic（自动） |
| lessons | Agent 复盘教训（你写的） | agent-review skill |
| journal | 操作日志（系统自动写） | auto_trader / trade_logic |
| pending_decisions | 你写，auto_trader 读和执行 | 你（agent-trade） |

## agent_candidates

worker 每轮刷新市场快照后，对热度榜代币调用 `evaluate_candidate()`，将 passed 的结果（tier=full/half）存入此表。extract 脚本直接读此表获取候选币，与面板"合约扫描与操作建议"数据同源。

```
round_number, token, data(JSON), tier, passed, hard_blocks(JSON)
```
- `data` JSON 即为候选币全字段（含 snapshot + analysis + 热度）
- extract 脚本取最近 2 轮，按 token 去重

## token_heat_history

```
token, score, mentions, unique_posts, round_number, recorded_at
```
- worker 每轮写入，供 `compute_composite_scores` 计算历史加权分
- Agent 不直接读此表

## market_snapshots

```
token (PK), snapshot(TEXT/JSON), analysis(TEXT/JSON), updated_at
```

snapshot JSON 字段详见 `references/snapshot字段说明.md`。worker 每轮刷新。

## trade_positions

活跃持仓条件：`status IN ('OPEN', 'PARTIAL')` 且 `json_extract(signal_snapshot, '$.source') = 'agent'`

PARTIAL 表示已部分止盈（TP1 触发），剩余仓位仍在运行，仍占用保证金。

## pending_decisions（你写入的）

必填字段：action, token, reason, status='pending'
开仓额外必填：tier, dimension_data, market_overview, lesson_checked, source_round, social_score
entry_price/stop_loss/tp1_price/tp2_price 由系统按实时价自动计算。平仓不由 Agent 处理，系统自动执行。

## lessons（agent-review 写入，你读取）

每次开仓前必须查。learned=0 的仍适用。severity: critical > warning > medium。
token='*' 为全局教训，适用于所有币种。
